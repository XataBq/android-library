## Educational goal

Teach the learner to reason about UI-layer responsibilities and state-driven Android interfaces.

The topic must explain:

- the purpose of the UI layer;
- the difference between UI elements and state holders;
- UI state as a representation of what the interface should display;
- state production and state consumption;
- user actions and UI events;
- Unidirectional Data Flow;
- immutable state;
- event handling;
- state restoration boundaries;
- lifecycle-aware state observation;
- derived UI state;
- loading, content, empty and error states;
- one-time actions and durable state;
- why UI framework components should not coordinate application data directly.

The topic must focus on architecture and responsibility boundaries rather than framework-specific API usage.

## Required conceptual model

The topic must use this conceptual flow:

```text
User action
    ↓
UI event
    ↓
State holder
    ↓
Application or data operation
    ↓
Updated state
    ↓
UI rendering

The text must explain that this is a conceptual responsibility flow, not a mandatory class diagram or library-specific implementation.

Required distinctions

The topic must clearly distinguish:

UI elements

Examples:

Activity;
Fragment;
Compose UI;
custom View;
reusable UI component.

Responsibilities include:

rendering state;
capturing user interaction;
forwarding actions;
interacting with framework-owned UI APIs.
State holders

Responsibilities include:

producing UI state;
coordinating UI-related operations;
transforming application data into displayable state;
enforcing UI-level decision logic;
surviving configuration changes where appropriate.

The topic may use ViewModel as the primary Android example, but must not reduce the general state-holder concept to ViewModel alone.

UI state

UI state must be described as data representing what the UI should display at a specific moment.

The topic must distinguish:

screen state;
source data;
derived state;
local element state;
state that must survive configuration changes;
state that must survive process recreation;
ephemeral interaction details.
Events and effects

The topic must distinguish:

durable state;
user actions;
state transitions;
framework interactions;
one-time UI effects.

The topic must not present a single event-wrapper mechanism as universally correct.

It must explain that navigation, messages and external framework actions require explicit ownership and delivery semantics.

Required examples

Use concise Kotlin examples where useful.

Examples should illustrate concepts such as:

data class ProfileUiState(
    val isLoading: Boolean,
    val userName: String?,
    val errorMessage: String?,
)

The topic should also discuss why several independent nullable flags can create invalid combinations.

At least one example must use a sealed hierarchy or another explicit state model, such as:

sealed interface ProfileUiState {
    data object Loading : ProfileUiState
    data class Content(val userName: String) : ProfileUiState
    data object Empty : ProfileUiState
    data class Error(val message: String) : ProfileUiState
}

The content must compare the trade-offs between:

one data class with independent properties;
mutually exclusive state variants.

It must not claim one representation is always superior.

Practice requirements

practice.md must contain at least four exercises.

Exercise 1 — Responsibility classification

Classify responsibilities between:

UI element;
state holder;
Data layer;
framework interaction.
Exercise 2 — Invalid state combinations

Analyze a UI model containing independent booleans and nullable values.

Identify impossible or contradictory states.

Propose a safer state model.

Exercise 3 — Broken UDF

Review code where:

the UI mutates shared state directly;
rendering triggers network requests;
state is updated from several unrelated locations.

Identify violations and redesign the responsibility flow.

Exercise 4 — State versus effect

Classify examples such as:

loading indicator;
displayed data;
validation error;
snackbar;
navigation;
text field input;
selected tab;
file picker launch.

The learner must justify ownership and delivery semantics.
