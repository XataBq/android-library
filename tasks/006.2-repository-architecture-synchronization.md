## 4. Create `docs/COMPETENCY_NORMALIZATION_WORKFLOW.md`

Create a reusable author/reviewer workflow for normalizing source items into canonical competencies.

This document must complement, not duplicate, the source import workflow.

Required sections:

### 4.1 Purpose

Explain that normalization begins only after source extraction and review-quality evidence exist.

Normalization converts source evidence into repository-owned canonical capabilities.

### 4.2 Inputs

Required inputs:

- one reviewed source package;
- the complete existing canonical registry;
- the canonical competency model;
- source-item text and locators;
- existing normalization reports.

### 4.3 Decision sequence

For each source item or coherent source-item group:

1. Identify the demonstrable capability supported.
2. Search the entire canonical registry.
3. Classify the candidate as:
   - exact semantic match;
   - partial overlap;
   - distinct capability;
   - insufficient or unsuitable evidence.
4. For an exact match, add evidence to the existing competency.
5. For partial overlap, determine whether the source is narrower, broader, or contains multiple capabilities.
6. Create a new competency only when no existing competency expresses the same demonstrable capability.
7. Defer evidence that does not justify a stable competency.
8. Document non-trivial merge, split, reuse, and rejection decisions.

### 4.4 Review rules

Include:

- no inferred competencies;
- source packages are immutable during normalization;
- canonical wording is source-independent;
- one primary capability per competency;
- evidence does not define identity;
- different sources do not justify duplicate competencies;
- semantic duplicate review is manual;
- all evidence references are versioned;
- status remains `review` until human approval.

### 4.5 Cross-source normalization

Explain the expected behavior when a second source is imported:

- compare against all existing competencies;
- attach evidence to existing competencies where equivalent;
- revise canonical wording only when meaning remains stable and broader evidence warrants it;
- avoid source-specific wording;
- record conflicts or materially different claims;
- do not silently weaken or strengthen recommendations.

### 4.6 Output package

Document the current package form:
