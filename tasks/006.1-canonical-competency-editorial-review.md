# Track A — Editorial review

## 2. Review every pilot competency

Review all pilot competencies individually.

For each competency verify:

1. It expresses one independently demonstrable capability.
2. Its title is action-oriented.
3. Its title and outcome use the same primary capability.
4. Its outcome is source-independent.
5. Its outcome does not exceed the evidence.
6. Its outcome does not combine unrelated capabilities.
7. Its evidence set is coherent.
8. Any merge is semantically necessary.
9. Any source-item reuse genuinely supports multiple independent competencies.
10. An equivalent pilot competency does not already exist.
11. Its ID is stable, concise, and source-independent.
12. Its normalization notes accurately explain the editorial decision.

## 3. Resolve the identified review concerns

Pay special attention to the following current pilot areas.

### 3.1 Lifecycle competency

Current candidate:

```text
analyze-android-component-lifecycle-constraints
```

Re-evaluate:

- whether `Analyze` is the clearest observable verb;
- whether lifecycle, component recreation, process termination, state ownership, and data ownership form one coherent capability;
- whether `keep-app-components-independent` is needed as evidence here;
- whether this competency overlaps with the separate component-design competency.

Prefer a narrower explanatory capability when it is better supported, for example:

```text
Explain lifecycle constraints of Android components
```

This example is guidance, not a mandatory final wording.

### 3.2 Independent component design competency

Current candidate:

```text
design-independent-app-components
```

Re-evaluate:

- whether `short-lived coordinators` is explicitly supported by the imported evidence;
- whether `self-contained` conflicts with the statement that components obtain data from other components;
- whether reuse of `keep-app-components-independent` is semantically justified after the lifecycle competency is refined;
- whether the outcome is too broad.

### 3.3 Separation of concerns competency

Current candidate:

```text
apply-separation-of-concerns
```

It currently combines evidence related to:

- architecture boundaries;
- separation of concerns;
- module responsibility boundaries.

Verify that these source items support one capability rather than several adjacent capabilities.

Do not merge them merely because they belong to the same architectural theme.

Either:

- retain the merge and strengthen the normalization rationale; or
- narrow the evidence set; or
- split the competency only if the source clearly supports independently demonstrable capabilities.

### 3.4 Persistent data models competency

Current candidate:

```text
explain-persistent-data-models
```

Check whether the outcome combines too many distinct benefits:

- independence from UI lifecycle;
- resilience to component recreation;
- process removal;
- intermittent network access.

Keep only claims explicitly supported by the linked source items.

### 3.5 Single source of truth competency

Review for outcome length and scope.

Avoid combining:

- ownership;
- mutation centralization;
- exposure;
- source selection;
- offline behavior;

unless all are inseparable parts of one coherent competency and directly supported by the evidence.

### 3.6 Repository responsibilities competency

Current candidate:

```text
explain-repository-responsibilities
```

Verify whether one competency can coherently include:

- exposing application data;
- centralizing changes;
- resolving conflicts;
- hiding data sources;
- coordinating sources;
- defining source responsibilities.

If retained, simplify the outcome without dropping essential meaning.

### 3.7 Optional domain layer competency

Current candidate:

```text
evaluate-optional-domain-layer
```

Review whether this currently merges two distinct capabilities:

- deciding when a domain layer is justified;
- designing each use case as single-purpose.

If they are independently assessable, do not force them into one competency.

However, do not increase the pilot beyond 12 competencies merely to preserve every source item. It is acceptable to leave one source item uncovered and document the decision.

### 3.8 Dependency injection competency

Current candidate:

```text
apply-dependency-injection
```

Verify:

- whether `verify dependencies` is actually supported;
- whether generalizing from a Hilt-specific recommendation to dependency injection remains faithful;
- whether framework-managed classes are within the imported evidence;
- whether the outcome uses one primary action.

Do not retain claims merely because they are generally true about DI.

## 4. Wording rules for the revision

Use observable verbs.

Preferred:

```text
Explain
Describe
Compare
Design
Apply
Evaluate
Identify
Justify
```

Use `Analyze` only when the evidence and expected demonstration genuinely require analysis.

Avoid vague or evaluative language such as:

```text
understand
know
learn
appropriate
proper
best
clean
robust
well-designed
```

unless the term has an explicit operational meaning in the evidence.

Outcomes should normally be one sentence and approximately 15–35 words.

Longer outcomes are allowed only when shortening them would remove an essential supported distinction.

Do not optimize wording mechanically by word count.

## 5. ID policy

Rename competency IDs only when the current ID is:

- tied to wording that is being materially changed;
- unnecessarily long;
- misleading about competency scope;
- source-specific;
- inconsistent with the revised title.

Do not rename stable IDs for cosmetic preference alone.

When an ID changes:

- update all references;
- update the normalization review report;
- update fixtures and tests if they use that ID;
- confirm that no committed external reference exists yet.

Because Task 006 is not committed, compatibility aliases are not required.

## 6. Competency count


Do not increase the count simply to avoid difficult merge decisions.

Do not reduce the count by merging unrelated capabilities.

# Track B — Technical review

## 8. Review both canonical schemas

Verify that schemas enforce the initial architecture model exactly.

Required checks include:

### Competency set schema

- required fields;
- `additionalProperties: false`;
- non-empty `id`, `title`, and `description`;
- valid language format according to repository conventions;
- positive integer `version`;
- permitted status values;
- correct `schema_version`.

### Competencies schema

- required top-level fields;
- `additionalProperties: false`;
- non-empty competencies array;
- required competency fields;
- non-empty title and outcome after schema-level validation where possible;
- non-empty evidence array;
- evidence required fields;
- non-empty `item_ids`;
- unique `item_ids` where supported by JSON Schema;
- `additionalProperties: false` at every object level;
- optional `normalization_notes`;
- no future-model fields.

Do not add semantic concepts that are outside Task 006.

## 9. Review validator behavior

Verify `scripts/validate_competencies.py` against the architecture and Task 006 acceptance criteria.

It must correctly enforce:

1. Source package schema validation.
2. Canonical package schema validation.
3. Competency set ID agreement.
4. Competency set version agreement.
5. Canonical competency ID uniqueness across all canonical packages.
6. Referenced source package existence.
7. Exact source version agreement.
8. Referenced item existence.
9. Non-empty evidence.
10. Non-empty item lists.
11. Duplicate item IDs within one evidence entry.
12. Exact duplicate evidence entries.
13. Unsupported fields through schema validation.
14. Independent validity of source packages.
15. Clear non-zero exit behavior when validation fails.

## 10. Validator robustness

Inspect for the following failure modes:

- validator crash on malformed YAML;
- unhandled missing file;
- ambiguous package discovery;
- directory name silently treated as canonical identity;
- duplicate package IDs across directories;
- duplicate source IDs across directories;
- invalid source package contaminating reference lookup;
- non-deterministic error ordering;
- platform-dependent path output;
- accidental acceptance of booleans where integers are expected;
- duplicate evidence detection depending on item order;
- silent skipping of unknown files;
- one failure preventing useful diagnostics for remaining packages.

Fix only concrete issues found in the current implementation.

Do not rewrite the validator without need.

## 11. Duplicate evidence semantics

Exact duplicate evidence entries should be treated as duplicates even when `item_ids` are listed in a different order, because evidence item order has no semantic meaning.

Example:

```yaml
- source_id: source-a
  source_version: 1
  item_ids: [item-a, item-b]

- source_id: source-a
  source_version: 1
  item_ids: [item-b, item-a]
```

The validator should reject this as duplicate evidence.

If the current implementation already does so, retain it and confirm with a test.
